# SPEC: ассистент на сайте бизнес-тренера (терминал в браузере)

Документ для разработки бэкенда и интеграций. Поведение диалога задаётся файлом `prompt_system.txt`. Факты о тренингах — в каталоге `data/*.txt`; при сборке запроса приложение подставляет их в сообщение с пометкой «База знаний» (или отдельным системным блоком).

## Артефакты

| Файл | Назначение |
|------|------------|
| `prompt_system.txt` | Системный промпт для LLM (без инструкций про логи и инфраструктуру). |
| `schema.sql` | Таблицы сессий, сообщений и лидов. |
| `EnvExample` | Переменные окружения (шаблон). |
| `load_env.py` | Вызов `load_env()` в точке входа до чтения `os.environ`. |

## Провайдер LLM: ProxyAPI

Используется OpenAI-совместимый API ProxyAPI.

- Базовый URL: `https://openai.api.proxyapi.ru/v1` ([документация](https://proxyapi.ru/docs/openai-compatible-api)); в окружении — `OPENAI_BASE_URL`.
- Авторизация: заголовок `Authorization: Bearer <OPENAI_API_KEY>`.
- Поле `model`: формат `провайдер/модель`, например `openai/gpt-4o-mini` (список актуальных имён — в личном кабинете и документации ProxyAPI).

Клиент: официальный SDK OpenAI с переопределением `base_url`, либо любой HTTP-клиент на `POST /v1/chat/completions`.

Рекомендации:

- Таймауты и повтор с backoff при 429/5xx.
- Лимит длины истории: последние N пар сообщений + системный промпт + блок базы знаний (или RAG-чанки).
- Логировать на сервере `request_id` из ответа (если есть), без логирования полного текста базы знаний в продакшене при желании сократить утечку данных.

## База знаний

На старте достаточно читать `.txt` из `data/` и подставлять в промпт целиком или частями, если умещается в контекст. При росте объёма — вынести chunking + эмбеддинги + векторный поиск; retrieval результаты подставлять в тот же блок «База знаний».

## Заявка (лид): поток данных

Цель: **Telegram — основное уведомление**, **БД — источник истины**, **email — опциональная копия**.

Рекомендуемый порядок при создании лида:

1. **Валидация** полей (телефон в минимально разумном формате, непустое имя по правилам продукта).
2. **INSERT** в таблицу `leads` со статусом `new`.
3. **Telegram** `sendMessage` в чат `TELEGRAM_LEAD_CHAT_ID`. При успехе — обновить `telegram_message_id`, статус `notified`.
4. При ошибке Telegram — записать `error_detail`, статус `failed` (или оставить `new` и ретраить по cron — на усмотрение).
5. Если `EMAIL_LEADS_ENABLED=true` — отправить письмо; при успехе выставить `email_sent_at`, при ошибке — `email_error` (лид всё равно считается созданным).

Идемпотентность: опционально не создавать дубликат лида с тем же `session_id` и тем же номером за короткий интервал (например 10 минут) — защита от двойного клика.

### Telegram

- Создать бота у `@BotFather`, получить `TELEGRAM_BOT_TOKEN`.
- Узнать `TELEGRAM_LEAD_CHAT_ID`: для личных сообщений пользователь пишет боту `/start`, затем запрос к `getUpdates` или сервисы вида «узнать chat id»; для группы — добавить бота в группу и взять id группы (часто отрицательный).
- Текст сообщения: имя, телефон, удобное время, тема, короткое резюме; при желании ссылка на админку с `lead id`.

### Email (опционально)

SMTP-параметры из `EnvExample`. Для Яндекс/Mail.ru обычно нужен пароль приложения. Для доставляемости с доменного адреса настройте SPF/DKIM на стороне почтового провайдера.

## Интеграция с моделью: надёжный сбор лида

Предпочтительно **function calling** (инструмент `submit_lead`) со схемой полей:

- `display_name` (string)
- `phone` (string)
- `preferred_contact_at` (string)
- `topic` (string, optional)
- `notes` (string, optional)

После ответа модели с `tool_calls` бэкенд выполняет сохранение и доставку, затем в диалог добавляется сообщение `role: tool` с результатом (`success`, `lead_id`). Только после этого модель генерирует финальное текстовое подтверждение пользователю.

Если используется модель без инструментов — fallback: эвристика или второй проход (NLU) по истории; менее надёжно.

## Сессии и история

- Каждому анонимному или авторизованному посетителю выдаётся `session_id` (UUID), хранится в cookie/localStorage и передаётся заголовком или телом запроса.
- Сообщения пользователя и ответы ассистента пишутся в `chat_messages` для контекста и разбора споров.
- В запрос к LLM передаётся скользящее окно последних сообщений (например 20–40 последних или ограничение по токенам).

## Логирование и тесты

- Серверные логи: `session_id`, укороченный текст пользователя, длина ответа, код ошибки провайдера — без полных PII в общий лог или с маскировкой телефона.
- «Терминал» на фронте может дублировать поток в консоль браузера только для режима отладки.
- Тест-кейсы из продуктовых требований: стоимость, программа тренинга, контакты, оффтоп, границы грубости, полный сценарий заявки с подтверждением записи в БД и одним сообщением в Telegram (тестовый чат).

## Распараллеливание

Пользовательский диалог последователен. Параллельно имеет смысл выполнять только независимые задачи (например retrieval + классификация намерения). Два одновременных ответа ассистента одному `session_id` не нужны.

## Миграции PostgreSQL

Файл `schema.sql` ориентирован на SQLite (`AUTOINCREMENT`, `datetime('now')`). Для PostgreSQL:

- Заменить типы времени на `TIMESTAMPTZ DEFAULT now()`.
- Заменить автоинкремент на `GENERATED ALWAYS AS IDENTITY` или `SERIAL`.
- Для `metadata_json` / `meta_json` использовать тип `JSONB` при желании.
